### **2. АКТУАЛИЗАЦИЯ (8 минути)**

**Преговор на Linux права (DAC):**
```bash
# Традиционни права
ls -la /var/www/html/
# -rw-r--r-- 1 www-data www-data 612 Jan 1 index.html

# Собственик, група, други
# rwx = read, write, execute

# Проблем: Ако процес работи като www-data,
# той има достъп до ВСИЧКО, което www-data притежава
```

**DAC ограничения:**
```
Проблем с DAC:
├── Процесът наследява правата на потребителя
├── Root има пълен достъп до всичко
├── Компрометиран процес = компрометиран потребител
└── Няма ограничение ЗА КАКВО се използват правата

Пример: Apache (www-data) може да:
├── Чете всички файлове на www-data
├── Пише навсякъде, където www-data може
└── Изпълнява произволни команди
```

### **3. НОВ МАТЕРИАЛ (35 минути)**

#### **Част 1: MAC концепция (10 минути)**

**MAC vs DAC:**
```
┌─────────────────────────────────────────────────────────────┐
│                    DAC vs MAC                               │
├─────────────────────────────────────────────────────────────┤
│ DAC (Discretionary)         │ MAC (Mandatory)               │
│ • Собственикът решава       │ • Системата решава            │
│ • Права: rwx                │ • Политики определят достъпа  │
│ • Лесно за управление       │ • По-сложно за конфигуриране  │
│ • По-малко сигурно          │ • Много по-сигурно            │
│ • chmod, chown              │ • SELinux, AppArmor           │
└─────────────────────────────────────────────────────────────┘

MAC принцип:
"Процес може да прави САМО това, което е ИЗРИЧНО РАЗРЕШЕНО"
```

**Defense in Depth:**
```
┌─────────────────────────────────────────────────────────────┐
│                   СЛОЕВЕ НА ЗАЩИТА                          │
├─────────────────────────────────────────────────────────────┤
│  Firewall (iptables/nftables)                               │
│  ─────────────────────────────────────                      │
│  Мрежова защита (SSH keys, fail2ban)                       │
│  ─────────────────────────────────────                      │
│  MAC (AppArmor/SELinux)           ← ДНЕШНАТА ТЕМА          │
│  ─────────────────────────────────────                      │
│  DAC (chmod/chown права)                                    │
│  ─────────────────────────────────────                      │
│  Приложение                                                 │
└─────────────────────────────────────────────────────────────┘
```

#### **Част 2: AppArmor в Ubuntu (15 минути)**

**AppArmor основи:**
```
AppArmor = Application Armor

Характеристики:
├── Включен по подразбиране в Ubuntu
├── Path-based (базиран на пътища)
├── По-лесен за разбиране от SELinux
├── Профили за всяко приложение
└── Два режима: enforce и complain

Профил определя:
├── Какви файлове може да чете/пише
├── Какви мрежови операции може да прави
├── Какви capabilities може да използва
└── Какви други програми може да изпълнява
```

**Практика 2.1: Проверка на AppArmor статус**
```bash
# Статус на AppArmor
sudo aa-status

# Резултат показва:
# - Заредени профили
# - Профили в enforce режим
# - Профили в complain режим
# - Процеси с профили

# Проверка дали AppArmor е активен
sudo systemctl status apparmor

# Kernel параметри
cat /sys/module/apparmor/parameters/enabled
# Y = активен
```

**Практика 2.2: Преглед на профили**
```bash
# Профилите се намират в:
ls /etc/apparmor.d/

# Примерни профили:
# usr.sbin.mysqld    - MySQL сървър
# usr.sbin.cups      - Print сървър
# usr.bin.firefox    - Firefox браузър

# Преглед на профил
sudo cat /etc/apparmor.d/usr.sbin.mysqld

# Структура на профил:
# ─────────────────────────────────────────
# /usr/sbin/mysqld {
#   # Включване на абстракции
#   #include <abstractions/base>
#
#   # Права за файлове
#   /var/lib/mysql/ r,
#   /var/lib/mysql/** rwk,
#
#   # Мрежови права
#   network tcp,
#
#   # Capabilities
#   capability setuid,
# }
```

**Практика 2.3: AppArmor режими**
```bash
# Превключване на профил в complain режим
# (логва нарушения, но не блокира)
sudo aa-complain /usr/sbin/mysqld

# Превключване в enforce режим
# (блокира нарушенията)
sudo aa-enforce /usr/sbin/mysqld

# Деактивиране на профил
sudo aa-disable /usr/sbin/mysqld

# Презареждане на профили
sudo systemctl reload apparmor
```

**Практика 2.4: Създаване на прост профил**
```bash
# Инсталиране на инструменти
sudo apt install apparmor-utils -y

# Автоматично генериране на профил
sudo aa-genprof /usr/bin/ping

# Следвайте инструкциите:
# 1. В друг терминал изпълнете командата
# 2. Натиснете 'S' за сканиране
# 3. Одобрете или откажете всяко действие
# 4. Натиснете 'F' за край

# Преглед на генерирания профил
sudo cat /etc/apparmor.d/usr.bin.ping
```

**Практика 2.5: Преглед на логове**
```bash
# AppArmor логове в:
sudo dmesg | grep apparmor
sudo journalctl -xe | grep apparmor

# Или в audit.log (ако е наличен)
sudo cat /var/log/audit/audit.log | grep apparmor

# Пример за DENIED съобщение:
# apparmor="DENIED" operation="open" 
# profile="/usr/sbin/mysqld" name="/etc/passwd" 
```

#### **Част 3: SELinux основи (10 минути)**

**SELinux концепция:**
```
SELinux = Security-Enhanced Linux

Характеристики:
├── Разработен от NSA
├── Използва се в RHEL, CentOS, Fedora
├── Label-based (базиран на етикети)
├── По-мощен, но по-сложен от AppArmor
└── Три режима: enforcing, permissive, disabled

Контексти (етикети):
user:role:type:level
Пример: system_u:object_r:httpd_sys_content_t:s0
```

**SELinux команди (демонстрация):**
```bash
# Проверка на статус
getenforce
# Enforcing, Permissive, или Disabled

sestatus
# Подробна информация

# Временна промяна на режим
sudo setenforce 0    # Permissive
sudo setenforce 1    # Enforcing

# Постоянна промяна
sudo nano /etc/selinux/config
# SELINUX=enforcing
# SELINUX=permissive
# SELINUX=disabled

# Преглед на контексти
ls -Z /var/www/html/
# system_u:object_r:httpd_sys_content_t:s0 index.html

# Промяна на контекст
sudo chcon -t httpd_sys_content_t /var/www/html/newfile.html

# Възстановяване на контексти
sudo restorecon -Rv /var/www/html/
```

**SELinux Booleans:**
```bash
# Списък на booleans
getsebool -a | grep httpd

# Разрешаване на httpd да изпраща email
sudo setsebool -P httpd_can_sendmail on

# Разрешаване на home directories
sudo setsebool -P httpd_enable_homedirs on
```

### **5. ОБОБЩЕНИЕ (8 минути)**

**Сравнение AppArmor vs SELinux:**
```
┌─────────────────────────────────────────────────────────────┐
│                AppArmor vs SELinux                          │
├──────────────────┬──────────────────────────────────────────┤
│ AppArmor         │ SELinux                                  │
├──────────────────┼──────────────────────────────────────────┤
│ Path-based       │ Label-based                              │
│ По-лесен         │ По-мощен                                 │
│ Ubuntu, SUSE     │ RHEL, CentOS, Fedora                    │
│ Profiles         │ Policies                                 │
│ aa-status        │ sestatus                                 │
│ enforce/complain │ enforcing/permissive                     │
└──────────────────┴──────────────────────────────────────────┘

Кога да използвате MAC:
✓ Публични сървъри (web, mail, DNS)
✓ Критични бизнес приложения
✓ Многопотребителски системи
✓ Compliance изисквания (PCI-DSS, HIPAA)
```

### ПРИЛОЖЕНИЕ 2: ПРИМЕРЕН APPARMOR ПРОФИЛ

```bash
# /etc/apparmor.d/usr.sbin.example-app

#include <tunables/global>

/usr/sbin/example-app {
  # Включване на базови абстракции
  #include <abstractions/base>
  #include <abstractions/nameservice>
  
  # Изпълними файлове
  /usr/sbin/example-app mr,
  
  # Конфигурационни файлове (само четене)
  /etc/example-app/ r,
  /etc/example-app/** r,
  
  # Данни (четене и писане)
  /var/lib/example-app/ rw,
  /var/lib/example-app/** rwk,
  
  # Логове
  /var/log/example-app/ rw,
  /var/log/example-app/** rw,
  
  # PID файл
  /run/example-app.pid rw,
  
  # Мрежови права
  network tcp,
  network udp,
  
  # Capabilities
  capability net_bind_service,
  capability setuid,
  capability setgid,
  
  # Забранен достъп (изрично)
  deny /etc/shadow r,
  deny /etc/passwd w,
  deny /root/** rwx,
}
```
